# 营收到卡轻量核算工具需求文档

## 1. 项目背景

交付老师和品牌方在核对门店收入时，经常需要解释不同平台的营收何时实际到银行卡。由于不同平台存在不同账期，例如微信 T+1、支付宝 T+1 或特殊 T+2，品牌方容易混淆“营收日期”和“到账日期”。

本工具用于帮助用户手动录入各平台结算金额，并根据平台账期自动计算预计到卡日期，最终形成平台维度和整体维度的到账视图。

## 2. 产品定位

一个无需登录、无需上传账单、无需维护门店数据库的轻量网页工具。

核心价值：

```text
把多个平台的结算金额，按照各自账期，转换成清晰的预计到卡明细和整体到账汇总。
```

## 3. 目标用户

- 交付老师
- 品牌运营人员
- 品牌财务核对人员
- 需要向门店解释收入到账节奏的内部人员

## 4. MVP 范围

第一版优先支持：

- 微信
- 支付宝

默认账期：

- 微信：T+1
- 支付宝：T+1

第一版不做：

- 账单上传
- 登录注册
- 门店数据库
- 历史记录保存
- 平台 API 对接
- Excel / PDF 导出
- 手续费、扣款、补贴拆分计算

## 5. 主流程

```text
1. 选择本次核算的平台
2. 填写各平台营收周期与结算金额
3. 自动计算各平台预计到卡日期
4. 查看平台到卡明细
5. 查看整体到卡汇总
6. 按到账周期筛选查看
```

## 6. 功能需求

### 6.1 选择平台

用户可以勾选本次需要核算的平台。

第一版平台选项：

- 微信
- 支付宝

用户勾选平台后，页面自动生成对应的平台录入卡片。

### 6.2 填写平台数据

每个平台卡片需要填写：

- 营收开始日期
- 营收结束日期
- 结算金额
- 账期天数

系统默认带出平台账期，用户可手动调整。

示例：

```text
平台：微信
营收周期：2026-05-01 至 2026-05-01
结算金额：1000
账期：T+1
预计到卡日期：2026-05-02
```

### 6.3 自动计算到卡日期

计算规则：

```text
预计到卡日期 = 营收结束日期 + 账期天数
```

第一版按自然日计算，不跳过周末和节假日。

如果用户手动调整账期，系统需要实时重新计算预计到卡日期。

### 6.4 平台到卡明细

系统生成平台维度明细，展示：

- 平台名称
- 营收周期
- 结算金额
- 账期
- 预计到卡日期

### 6.5 整体到卡汇总

系统按照预计到卡日期聚合金额。

示例：

| 到卡日期 | 预计到卡金额 | 来源平台 |
|---|---:|---|
| 2026-05-02 | ¥1,800.00 | 微信、支付宝 |

如果多个平台同一天到账，需要合并展示总金额，并保留来源平台明细。

### 6.6 到账周期筛选

结果页支持按到账日期筛选。

筛选项：

- 到账开始日期
- 到账结束日期

筛选后，平台明细和整体汇总只展示该到账周期内的数据。

注意：到账周期筛选只影响结果展示，不改变原始计算。

## 7. 校验规则

- 未选择平台时，显示空状态提示。
- 结算金额不能为空。
- 结算金额不能小于 0。
- 营收结束日期不能早于营收开始日期。
- 账期天数不能小于 0。
- 日期或金额错误时，在对应输入项附近提示。

## 8. 页面结构

页面建议分为三块：

```text
录入设置
- 选择本次核算的平台
- 填写各平台营收周期与结算金额

到账计算
- 自动计算各平台预计到卡日期
- 支持手动调整账期

结果查看
- 平台到卡明细
- 整体到卡汇总
- 按到账周期筛选查看
```

## 9. 视觉风格

采用深色玻璃质感工作台风格：

- 半透明卡片背景
- 轻微 backdrop blur
- 细边框
- 柔和阴影
- 紫色到青色的强调渐变
- 平台卡片 hover 轻微放大
- 空状态使用轻量插画
- 字体使用 Inter 或系统无衬线字体

## 10. 验收标准

满足以下条件视为 MVP 完成：

- 用户可以选择微信、支付宝。
- 用户可以分别录入营收周期和结算金额。
- 系统可以根据 T+N 自动计算预计到卡日期。
- 用户可以手动调整账期。
- 页面可以展示平台到卡明细。
- 页面可以展示按到卡日期聚合的整体汇总。
- 用户可以按到账周期筛选结果。
- 页面无需后端即可运行。
- 刷新页面后数据清空。
- 桌面和移动端展示不出现明显错位或溢出。
